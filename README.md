## 📌 Project Overview

IntelliTrip is a multi-user AI travel planner where users register, log in, and receive a fully personalized day-by-day itinerary generated by an LLM. Users provide their destination, number of days, budget preference, and interests — the AI does the rest. The itinerary is fully editable: activities can be added, removed, or regenerated per day. Budget estimates and hotel suggestions are included automatically.

## 🛠 Tech Stack & Justification

| Technology | Purpose | Why |
|---|---|---|
| **Next.js 14** (Pages Router) | Framework & routing | Industry standard for React apps; SSR-ready, great DX |
| **Tailwind CSS** | Styling | Utility-first approach keeps styles co-located and consistent |
| **Lucide React** | Icons | Lightweight, tree-shakable, consistent icon set |
| **Axios** | HTTP client | Cleaner interceptors for auth headers and error handling vs fetch |
| **React Context** | Auth state | Lightweight global state without Redux overhead for this scope |

## 🏗 High-Level Architecture

```
Browser (Next.js)
      │
      ├── AuthContext  ←  JWT stored in localStorage
      │       │
      │       └── Axios instance (attaches Bearer token to every request)
      │
      ├── Pages (dashboard, trips/[id])
      │       │
      │       └── Components (TripCard, DashboardHeader, ConfirmModal...)
      │
      └── API Layer (lib/api.js)
              │
              └── Express REST API  →  MongoDB
```

- All API calls are centralized in `lib/api.js` — no fetch calls scattered in components
- Auth state is managed globally via `AuthContext` — protected pages redirect to `/login` if no user
- UI state (modals, loading, errors) lives locally in each page component

---

## 🔐 Authentication & Authorization

- Users register with name, email, and password
- Passwords are hashed with **bcryptjs** before storage
- On login, the server issues a signed **JWT** (7-day expiry)
- The token is stored in `localStorage` and attached to every API request via an Axios interceptor
- `AuthContext` verifies the token on mount by calling `GET /api/auth/me`
- All protected pages check for an authenticated user — unauthenticated users are redirected to `/login`
- The backend enforces ownership on every trip operation — users cannot access or modify other users' trips

## 🤖 AI Agent Design

The AI agent is powered by **Anthropic Claude** on the backend. When a user clicks "Generate Itinerary", the frontend calls `POST /api/trips/:id/generate`. The backend constructs a structured prompt using the trip's destination, days, budget type, and interests — then parses Claude's response into three structured outputs:

| Output | Description |
|---|---|
| `itinerary[]` | Day-by-day activities with time-of-day labels (Morning / Afternoon / Evening) |
| `budgetEstimate` | Cost breakdown across flights, accommodation, food, and activities in INR |
| `hotels[]` | Recommended stays with name, type (Budget / Mid-range / Luxury), and rating |

The frontend renders each output in a dedicated tab, keeping the UI focused and scannable.

## ✨ Creative Feature — Inline Itinerary Editing

Beyond just generating an itinerary, users can **edit it live without regenerating the whole trip:**

- ✏️ **Edit any activity** inline by clicking the pencil icon
- ➕ **Add a new activity** to any day
- 🗑️ **Remove an activity** that doesn't fit
- 🔄 **Regenerate a single day** with optional custom preferences (e.g. "more outdoor activities")

**Why this was built:** AI output is never perfect for every traveler. Giving users full control over the generated plan respects their autonomy and makes the tool genuinely useful — not just a one-shot generator. This bridges the gap between AI automation and personal customization.

## 🎨 Key Design Decisions & Trade-offs

| Decision | Reasoning | Trade-off |
|---|---|---|
| CSS variable design system | Single source of truth for colors and spacing; dark mode support | Slightly more setup than plain Tailwind |
| Pages Router over App Router | More stable, better documented for auth patterns | Misses React Server Components |
| JWT in localStorage | Simple implementation, works across tabs | Less secure than httpOnly cookies; acceptable for this scope |
| Reusable `ConfirmModal` | One component handles delete and logout confirmations | Slightly more props, but eliminates duplication |
| Stats computed in `DashboardHeader` | Keeps dashboard.jsx as a thin orchestrator | Header is less "pure" — does minor data processing |

## ⚠️ Known Limitations

- JWT is stored in `localStorage` — vulnerable to XSS. A production app should use `httpOnly` cookies
- Day regeneration uses a mock preview in the frontend — in production this should call a dedicated API endpoint
- No pagination on the dashboard — performance may degrade with a large number of trips
- No unit or integration tests included in this version
